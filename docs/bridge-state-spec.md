# CCX-WiZ Bridge State Machine Spec

Governing spec for the bridge rewrite. Speed and correctness are the two design goals — every design decision optimizes for one without sacrificing the other.

## 1. CCX Message Sources

Understanding what generates each message type is critical to the bridge design.

### BUTTON_PRESS (type 1) — the fast path

Generated by native Lutron interactions: keypad presses, pico presses, scene recalls, occupancy sensors. These are **programmed into devices** via Designer transfers — the device itself fires the BUTTON_PRESS, the processor does NOT fan out per-zone LEVEL_CONTROLs afterward.

The bridge resolves BUTTON_PRESS → preset lookup table → dispatch all zones simultaneously. This is the primary path for responsive lighting control.

### LEVEL_CONTROL (type 0) — the external path

Generated by external integrations: Lutron app, Home Assistant → LEAP → processor. The processor serializes one LEVEL_CONTROL per zone, ~300ms apart. An 8-zone command takes ~2.5s to fully transmit.

The bridge handles each LEVEL_CONTROL individually as it arrives — no batching, no waiting for the "group" to complete.

### DIM_HOLD / DIM_STEP (types 2/3) — raise/lower

Physical hold-and-release on picos or keypads. DIM_HOLD starts the ramp, DIM_STEP stops it.

### COMPONENT_CMD (type 40) — CCT/color component

Only observed when synthetic devices (via Nucleo) are on the Thread mesh. Carries CCT/group info as unicast to registered devices. The bridge does not need to handle this — BUTTON_PRESS + preset table provides the same info.

### Color-only LEVEL_CONTROL

LEVEL_CONTROL can arrive with colorXy but no level field (level absent from CBOR body key 0). This happens when a user adjusts color independently of brightness (e.g., setting color first, then adjusting brightness separately in the app). The bridge should update color state but preserve the current level — do not treat missing level as 0.

### Capture evidence

All verified via promiscuous 802.15.4 capture (nRF sniffer, channel 25, 2026-03-28):

| Capture | Trigger | Messages observed |
|---------|---------|-------------------|
| `ccx-button-promiscuous.pcapng` | Keypad scene button | BUTTON_PRESS + ACK only, zero LEVEL_CONTROL |
| `ccx-multizone-level.pcapng` | Lutron app multi-zone | LEVEL_CONTROL per zone, ~300ms apart |
| `plain-real-sunnata-multi-zone-level.pcapng` | Lutron app → 8 Sunnatas | 8x LEVEL_CONTROL, seq 23→30, ~310ms spacing |

### Thread multicast retransmissions

Every CCX message is multicast to ff03::1 and retransmitted ~15-28 times within ~170ms for reliability. All copies are byte-identical with the same sequence number. Deduplication (section 4) filters these.

### DEVICE_REPORT from real devices

Real Sunnata dimmers are selective about reporting — in the 8-zone Sunnata capture, only 2 of 8 devices sent DEVICE_REPORT back to the processor. The bridge should not try to match this behavior exactly; one report per bridged zone per command is fine.

## 2. Zone State Model

A single typed object per zone replaces the current 8-map sprawl:

```typescript
interface ZoneState {
  level: number;           // 0-100 percent
  colorMode: "cct" | "xy"; // explicit, never inferred
  cct: number | null;      // Kelvin, set when colorMode="cct"
  colorXy: [number, number] | null; // CIE xy * 10000, set when colorMode="xy"
  activity: ZoneActivity;  // current motion state
  dirty: boolean;          // needs WiZ send this tick
  reportPending: boolean;  // needs DEVICE_REPORT after activity settles
}

type ZoneActivity =
  | { type: "idle" }
  | { type: "fading"; startLevel: number; targetLevel: number;
      startCct: number | null; targetCct: number | null;
      colorXy: [number, number] | null;
      startTime: number; durationMs: number }
  | { type: "ramping"; direction: "raise" | "lower";
      startLevel: number; startTime: number };
```

All state for a zone lives here. No separate maps. `colorMode` is always explicit — when CCT is set, `colorXy` is nulled, and vice versa. No ambiguity.

Key changes from current design:
- **No timers in state** — the tick loop drives everything (section 7)
- **`dirty` flag** — marks zones needing WiZ send this tick
- **`reportPending`** — DEVICE_REPORT fires once when activity completes, not per-step
- **Fades use wall-clock time** (`startTime` + `durationMs`) instead of step counters — no drift

### State access

```typescript
zones: Map<number, ZoneState>
```

Lazy-initialized on first packet for a zone. Default:
```typescript
{ level: 0, colorMode: "cct", cct: null, colorXy: null,
  activity: { type: "idle" }, dirty: false, reportPending: false }
```

## 3. State Transitions

```
          LEVEL_CONTROL (fade=0)
          BUTTON_PRESS (instant)
    ┌──────────────────────────────────┐
    │                                  │
    v                                  │
  IDLE ──── LEVEL_CONTROL (fade>0) ──► FADING
    │       BUTTON_PRESS (with fade)       │
    │                                      │ fade completes
    │       ┌──────────────────────────────┘
    │       v
    │     IDLE
    │
    ├──── DIM_HOLD ──────────────────► RAMPING
    │                                      │
    │       ┌──────────────────────────────┘ DIM_STEP / hit limit
    │       v
    └──── IDLE
```

### Transition rules

| Current | Event | Action |
|---------|-------|--------|
| IDLE | LEVEL_CONTROL fade=0 | Set level+color, mark dirty, stay IDLE, set reportPending |
| IDLE | LEVEL_CONTROL (color only, no level) | Update color state, preserve current level, mark dirty |
| IDLE | LEVEL_CONTROL fade>0 | Enter FADING, mark dirty (first step computed on next tick) |
| IDLE | BUTTON_PRESS | Dispatch each zone per preset table |
| IDLE | DIM_HOLD | Enter RAMPING, mark dirty |
| FADING | LEVEL_CONTROL (same target) | **Keep fading** — idempotent, don't restart |
| FADING | LEVEL_CONTROL (different target) | Cancel fade, handle as from IDLE |
| FADING | DIM_HOLD | Cancel fade, enter RAMPING |
| FADING | tick | Interpolate level+color from wall clock, mark dirty |
| FADING | tick (time >= duration) | Set final level, mark dirty, enter IDLE, set reportPending |
| RAMPING | DIM_STEP | Stop ramp, set final level, mark dirty, enter IDLE, set reportPending |
| RAMPING | LEVEL_CONTROL | Cancel ramp, handle as from IDLE |
| RAMPING | tick | Compute level from wall clock, mark dirty |
| RAMPING | tick (hit limit) | Auto-stop, same as DIM_STEP |

### Interrupts always win

Any new command with a different target cancels in-progress activity. If a command targets the same level a fade is already heading toward, the fade continues — this is idempotency, not suppression.

## 4. No Echo Suppression Needed

**Verified by promiscuous 802.15.4 capture (2026-03-28).** RA3 does NOT send per-zone LEVEL_CONTROL echoes after a BUTTON_PRESS. Native scene recalls (keypad, pico, occupancy) use BUTTON_PRESS only — devices execute presets locally. The previous bridge's fade absorption logic (0.5% float threshold) was solving a phantom problem.

LEVEL_CONTROL is only generated by external integrations (Lutron app, Home Assistant) and arrives one-per-zone with ~300ms spacing. These are real commands, not echoes — each should be dispatched immediately.

## 5. Deduplication

Dedup handles Thread multicast retransmissions — the same packet broadcast ~15-28 times for reliability. Each copy is byte-identical with the same sequence number.

```typescript
key: `${msgType}:${identifyingField}:${sequence}`
window: 200ms
```

Where `identifyingField` is:
- LEVEL_CONTROL: `zoneId`
- BUTTON_PRESS: `presetId`
- DIM_HOLD/DIM_STEP: `zoneId` (or `presetId` if zone=0)

200ms window. Thread retransmissions arrive within ~170ms. The current 2000ms window was conflating dedup with (unnecessary) echo suppression.

Cleanup: evict entries older than window on every check. Map stays small.

## 6. Dispatch Contract

`dispatch()` receives a fully-resolved command. No state inference inside dispatch.

```typescript
interface DispatchCommand {
  zoneId: number;
  level: number | null;    // null = color-only update, preserve current level
  colorMode: "cct" | "xy";
  cct: number | null;      // resolved (warm dim curve already evaluated)
  colorXy: [number, number] | null;
  fade: number;            // quarter-seconds, 0 = instant
  source: string;          // for logging: "LEVEL", "PRESET(Evening)", "RAMP"
}
```

### Resolution happens before dispatch

The packet handler resolves everything:
- Warm dim curve → concrete CCT value
- Missing CCT → pull from zone state or pairing default
- Color mode → explicit "cct" or "xy"
- Missing level → `null` (color-only command, preserve current brightness)

Dispatch only does: update zone state → set dirty flag → manage activity. No conditionals about what color data is present. No WiZ sends — that's the tick loop's job.

### WiZ send contract

`sendWiz()` receives a resolved output:

```typescript
interface WizCommand {
  ips: string[];
  port: number;
  level: number;
  colorMode: "cct" | "xy" | "off";
  cct: number | null;
  colorXy: [number, number] | null;
  cctTable: CctPoint[] | undefined;  // pre-resolved per pairing
}
```

No table lookups, no fallback logic inside sendWiz. Caller provides everything.

## 7. Unified Tick Loop

Replace per-zone `setInterval` timers with a single dispatch loop.

### Why

Current design: each fading/ramping zone gets its own `setInterval`. Problems:
- N-zone preset = N independent timers, drifting independently
- No coordination — zones in the same preset send WiZ at slightly different times
- Can't batch UDP sends
- Timer drift compounds per zone over long fades

### Design

```typescript
const TICK_MS = 50;  // 20 Hz tick rate

function tick() {
  const now = Date.now();
  const batch: WizCommand[] = [];

  for (const [zoneId, zone] of zones) {
    // Advance active animations
    if (zone.activity.type === "fading") {
      advanceFade(zone, now);
    } else if (zone.activity.type === "ramping") {
      advanceRamp(zone, now);
    }

    // Collect dirty zones for batched send
    if (zone.dirty) {
      zone.dirty = false;
      batch.push(buildWizCommand(zoneId, zone));
    }

    // Emit DEVICE_REPORT for zones that just settled
    if (zone.reportPending && zone.activity.type === "idle") {
      zone.reportPending = false;
      sendDeviceReport(zoneId, zone.level);
    }
  }

  // Send all WiZ commands for this tick
  for (const cmd of batch) {
    sendWiz(cmd);
  }
}
```

### Tick rate: 50ms (20 Hz)

- WiZ bulbs accept commands at ~30 Hz max (33ms). 50ms is safely under that.
- Matches human flicker perception threshold (~60 Hz for flicker, but dimming transitions are smooth at 20 Hz).
- Half the current ramp interval (100ms) → smoother ramps.
- Lower than the current fade step (250ms) → can interpolate sub-quarter-second for smoother fades.

### Fade interpolation (wall-clock, not step-counting)

```typescript
function advanceFade(zone: ZoneState, now: number) {
  const fade = zone.activity as FadingActivity;
  const elapsed = now - fade.startTime;
  const t = Math.min(1, elapsed / fade.durationMs);

  zone.level = fade.startLevel + t * (fade.targetLevel - fade.startLevel);
  if (fade.startCct != null && fade.targetCct != null) {
    zone.cct = Math.round(fade.startCct + t * (fade.targetCct - fade.startCct));
  }
  zone.dirty = true;

  if (t >= 1) {
    zone.level = fade.targetLevel;  // exact final value
    zone.activity = { type: "idle" };
    zone.reportPending = true;
  }
}
```

No step counter. No accumulated rounding error. Wall clock is the source of truth. The tick loop evaluates where each fade should be *right now* — if a tick is late, the interpolation catches up automatically.

### Ramp interpolation

```typescript
function advanceRamp(zone: ZoneState, now: number) {
  const ramp = zone.activity as RampingActivity;
  const elapsedSec = (now - ramp.startTime) / 1000;
  const delta = elapsedSec * RAMP_RATE_PCT_PER_SEC;

  if (ramp.direction === "raise") {
    zone.level = Math.min(100, ramp.startLevel + delta);
  } else {
    zone.level = Math.max(0, ramp.startLevel - delta);
  }

  // Re-evaluate warm dim CCT each tick
  if (pairing.warmDimCurve && zone.level > 0) {
    zone.cct = evalWarmDimCurve(getCurve(pairing.warmDimCurve), zone.level);
  }

  zone.dirty = true;

  // Auto-stop at limits
  if (zone.level >= 100 || zone.level <= 0) {
    zone.activity = { type: "idle" };
    zone.reportPending = true;
  }
}
```

### Benefits of the tick loop

1. **Synchronized preset fades** — all zones in a preset compute their state in the same tick, WiZ commands sent together
2. **No timer management** — no `setInterval`/`clearInterval` per zone, no cleanup bugs
3. **Natural batching** — dirty flags collect, one pass sends everything
4. **DEVICE_REPORT coalescing** — reports fire in the same tick pass where activity settles, naturally grouped
5. **Constant overhead** — one timer regardless of zone count

## 8. DEVICE_REPORT Behavior

### How real Lutron devices report

Real Sunnata dimmers are selective — in an 8-zone multi-level command, only 2 of 8 devices sent DEVICE_REPORT back to the processor. The bridge does not need to match this behavior exactly.

### Current bridge behavior (broken)

- Sends 6 DEVICE_REPORTs at 500ms intervals for a single scene recall
- Reports fire per-zone immediately rather than after execution completes
- Excess volume: N zones = N reports sent sequentially with artificial spacing

### New behavior

**One DEVICE_REPORT per bridged zone per command, emitted when activity settles to IDLE.**

The `reportPending` flag in ZoneState is set when a command arrives. It's only acted on when `activity.type === "idle"` — meaning the fade/ramp completed. The tick loop checks this naturally.

For instant commands (fade=0), the zone stays IDLE and reportPending fires on the next tick.

For fades, reportPending fires on the tick where the fade completes.

For ramps, reportPending fires on the tick where DIM_STEP stops the ramp.

### Sequence numbers on outgoing reports

The bridge maintains an 8-bit sequence counter for DEVICE_REPORT. Each report increments it. This is the bridge's own counter — unrelated to the incoming packet sequence.

## 9. WiZ Transport

### Single shared socket

One UDP socket for all WiZ communication. The current design already does this for setPilot but creates per-bulb sockets for CCT table fetch — fix that.

### CCT table fetch

On startup, fetch CCT tables using the shared socket:
- Send `getCctTable` to each unique IP sequentially (not parallel — one socket)
- 2s timeout per bulb, **1 retry** on timeout
- Log failures explicitly (not silently)
- Store per-IP; fall back to DEFAULT_CCT_TABLE with a warning

### Fire-and-forget sends

WiZ UDP is inherently fire-and-forget. No ack, no retry. The tick loop sends and moves on. If a bulb is unreachable, the packet is lost — this is by design (WiZ protocol has no reliability layer).

## 10. Logging Contract

Every silent failure in the current bridge becomes an explicit log:

| Event | Log level | Current behavior |
|-------|-----------|-----------------|
| Zone has no pairing | WARN | Silent skip |
| Decryption failed | DEBUG | Silent null return |
| Bulb CCT fetch timeout | WARN | Silent fallback |
| Dedup filtered | TRACE | No logging |
| Fade interrupted by new command | INFO | No logging |
| Fade completed | DEBUG | Not logged |
| Ramp auto-stopped at limit | INFO | Logged |
| DEVICE_REPORT sent | DEBUG | Logged |

## 11. Config Validation

Validate on startup, fail fast:

- Thread master key: 32 hex chars
- Thread channel: 11-26
- Pairings: each must have `zoneId` (number) and at least one WiZ IP (valid IPv4)
- Warm dim curve names: must exist in curve registry
- Sniffer device: path must exist (or be auto-detected)

Use runtime validation (Zod or manual checks). Invalid config = immediate exit with clear error message.

## 12. What This Spec Does NOT Cover

- **Frame pipeline / decryption**: separate concern, reusable as-is
- **Serial sniffer**: separate concern, but the 500ms shell race should be fixed independently
- **WiZ color math**: `wiz-color.ts` and `warm-dim.ts` are correct and well-tested, reuse as-is
- **Thread crypto**: solid, no changes needed

## Open Questions

1. **Ramp rate**: Is 4.75s (19 qs) the right full-range duration? Should this be configurable or derived from the Lutron system?

2. **Tick rate**: 50ms (20 Hz) proposed. May need tuning — too fast wastes CPU on idle systems, too slow makes ramps choppy. Could adaptive-tick: 50ms when zones are active, pause when all idle.

3. **WiZ command rate limit**: Do WiZ bulbs drop commands if sent faster than ~30 Hz? If so, the tick loop may need per-bulb rate limiting (skip sends if last send to that IP was <33ms ago).
