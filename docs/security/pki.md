# Lutron PKI & Certificate Architecture

Certificate chains, keys, and trust relationships across the Lutron ecosystem, documented from the iOS app, Lutron Designer .NET binary, processor TLS handshakes, and project database extraction.

## TL;DR: What's Shareable?

| Certificate | Shareable? | Why |
|------------|-----------|-----|
| `phoenix.crt` (lutron-root) | **Yes** — public CA | Shipped in every app and Designer install. Universal root. |
| `lutronelectronics.crt` (Caseta root) | **Yes** — public CA | Shipped in every app. Caseta LAP root. |
| Product chain certs (`radioRa3_products.crt`, etc.) | **Yes** — public CAs | Shipped in Designer. Intermediate CAs for all product lines. |
| Pegasus/Ketra trust chains | **Yes** — public bundles | Shipped in every app. 39 certs covering all device lines. |
| `revoked_certs.pem` | **Yes** — revocation list | Shipped in app. Blacklist of 6 compromised device certs. |
| `residential_client.pem` + key | **Universal** | Same cert/key in every Designer install. Shared secret. |
| `commercial_local_access.pfx` etc. | **Universal** | Same PFX in every Designer install. Shared secrets. |
| Your LEAP client certs (`lutron-ra3-cert.pem`) | **No** — per-processor | Generated during pairing, signed by YOUR processor's CA. |
| Designer project certs (`designer_client.crt`) | **No** — per-project | Generated by Designer for YOUR project database. |
| Processor server certs | **No** — per-processor | Unique to each physical processor (MAC in CN). |
| SSH certs (`ssh_designer-cert.pub`) | **No** — per-session | Short-lived (1 year), signed by project CA. |

## LEAP Protocol Headers

LEAP is **not HTTP**. It's newline-delimited JSON over raw TLS (mutual auth, port 8081). There are no HTTP headers, no `Server:` header, no cookies. The protocol is entirely:

```
→ {"CommuniqueType":"ReadRequest","Header":{"Url":"/server","ClientTag":"t1"}}\n
← {"CommuniqueType":"ReadResponse","Header":{"StatusCode":"200 OK","Url":"/server","ClientTag":"t1"},"Body":{...}}\n
```

### TLS Fingerprints

**RA3 Processor** (<ra3-ip>:8081):
- Protocol: **TLSv1.2** (not 1.3)
- Cipher: `ECDHE-ECDSA-CHACHA20-POLY1305`
- Server key: EC prime256v1 (P-256), 256-bit
- Temp key: X25519, 253-bit
- Secure renegotiation: supported
- ALPN: none
- Compression: none
- Certificate chain depth: 5 (leaf → processor-intermediate → devices → products → lutron-root)

**Caseta Bridge** (<caseta-ip>:8081):
- Protocol: **TLSv1.2**
- Cipher: `ECDHE-ECDSA-CHACHA20-POLY1305`
- Server key: EC prime256v1 (P-256), 256-bit
- Self-signed (no chain — bridge IS the CA)
- Accepts client CAs: itself + `Installer-SmartBridge<MAC>`

### Linux Earmarks

No definitive Linux signatures from TLS alone (no HTTP Server header). But:
- **TLSv1.2 with CHACHA20-POLY1305** is characteristic of OpenSSL 1.1+ or mbedTLS
- **X25519 key exchange** suggests modern OpenSSL (1.1.0+)
- **Secure Renegotiation support** is standard OpenSSL behavior
- The RA3 processor runs **Yocto Linux** (confirmed via firmware update package structure and SSH access)
- Caseta runs a **bare-metal/RTOS** stack (much simpler TLS, self-signed cert, no chain)

## Certificate Hierarchy

### Tree 1: Product Trust (Read-Only LEAP API)

This is the "who manufactured this device" hierarchy. Embedded in every app and Designer install. Used for port 8081 (product LEAP).

```
lutron-root (RSA 6144-bit, self-signed, 2016-2036)
├── radioRa3-products (RSA 4096-bit, 2020-2040)
│   └── radioRa3-devices (RSA 4096-bit, 2021-2040)
│       └── radioRa3-processors-027-4725-24 (RSA 4096-bit, 2021-2040)
│           └── radiora3-<MAC>-server (EC P-256, 20-year validity) ← per-processor leaf
├── athena-products (RSA 4096-bit, 2020-2040) ← Homeworks QSX
├── homeworksqs-products (RSA 4096-bit, 2019-2039)
├── quantum-products (RSA 4096-bit, 2019-2039)
├── myroom-products (RSA 4096-bit, 2019-2039)
└── pegasus-devices (intermediate for Ketra/CCX)
    ├── control-devices (intermediate for Thread/CCX control)
    │   ├── control_devices_027-5388-{01..16} ← per-factory-line device certs
    │   ├── control_devices_cb_027-5388-02 ← "cb" = circuit breaker?
    │   ├── control_devices_br_027-5388-01 ← "br" = bridge?
    │   └── control-devices-azure-{01,02} ← Azure IoT Hub provisioning
    └── ketra-devices (intermediate for Ketra bulbs/fixtures)
        ├── ketra_devices_027-5388-{01..16}
        ├── ketra_devices_cb_027-5388-02
        ├── ketra-lamp_BEAM_ls1627 ← Ketra A19/BR30 lamp
        ├── ketra_downlight_BEAM_ls1627 ← Ketra downlight
        ├── ketra_n3_BEAM_AUS37 ← Ketra N3 (Australian variant)
        └── ketra-devices-azure-{01,02}
```

### Tree 2: Caseta Local Access Protocol

Separate, simpler hierarchy. Each Caseta bridge is its own root CA.

```
SmartBridge<MAC> (EC P-256, self-signed, 2015-2035)
└── pylutron_caseta (client cert, issued during button-press pairing)
```

Also:
```
Caseta Local Access Protocol Cert Authority (RSA 2048-bit, self-signed, 2015-2035)
└── (client certs issued during pairing)
```

### Tree 3: Project SubSystem (Designer Write Access)

Per-project hierarchy. Generated when Designer creates/commissions a project. Used for WSS port 443 (full read-write LEAP).

```
Lutron Project SubSystem Certificate Authority (EC P-256, per-project, 2019+)
├── RadioRa3Processor<MAC> (EC P-256, 100-year validity!) ← processor server cert
├── DesignerClient (EC P-256, 10-year validity) ← Designer writes to processor
└── IPLServer<serial> (not yet observed) ← integration processor
```

### Tree 4: Designer Software Auth

Used for residential/commercial system access from Designer.

```
Lutron Designer Certificate Authority
└── Lutron Residential GUI (RSA 2048-bit, 2020-2039)
```

### Tree 5: Ketra/Pegasus Device Trust

For BLE commissioning of Ketra smart bulbs. Rotated annually.

```
pegasus_ketra_alpha_root (EC, per-year)
└── pegasus_ketra_alpha (EC, 1-year validity for 2020/2021, then 100-year for current)
```

```
orion_poc_root (EC, experimental)
└── orion_poc (EC, valid 2025-2026) ← next-gen product line?
```

### Generated Per-Pairing (client certs)

| File | Type | CN | Issuer | Custom OIDs | Notes |
|------|------|-----|--------|-------------|-------|
| `lutron-ra3-cert.pem` | Client cert | pylutron_caseta | RadioRa3Processor\<MAC\> | `1.3.6.1.4.1.40073.1.2`=000000000000, `.1.3`=pylutron_caseta | Signed by YOUR processor during pairing |
| `lutron-caseta-cert.pem` | Client cert | pylutron_caseta | SmartBridge\<MAC\> | same OIDs | Signed by YOUR bridge during pairing |

OID `1.3.6.1.4.1.40073` = Lutron Electronics (IANA Private Enterprise Number).
- `.1.2` = device serial number (000000000000 = generated by software pairing, not hardware)
- `.1.3` = device name / pairing identity

### SSH Access (Designer → Processor)

| File | Type | Identity | Signing CA | Validity | Principals | Notes |
|------|------|----------|------------|----------|------------|-------|
| `ssh_designer-cert.pub` | OpenSSH user cert | `designer@lutron` | project EC key | 1 year (2026-2027) | **root** | Designer gets root SSH to processors! |
| `ssh_designer.pub` | EC P-384 public key | `designer@lutron` | — | — | — | The key that gets signed |
| `project_ssh.pub` | EC P-384 public key | (project CA) | — | — | — | CA that signs SSH certs |

The project's EC key (`project_ec.pem`) is the signing authority for SSH certificates. Designer generates a keypair, has the project CA sign it as an SSH user certificate with `root` principal, and uses this to SSH into the processor for provisioning.

## How Certificates Are Generated

### 1. Factory (Lutron manufacturing)
- Each processor gets a unique server cert signed by product chain
  - `radiora3-<MAC>-server` signed by `radioRa3-processors-027-4725-24`
- Each Caseta bridge generates its own self-signed root CA (CN=SmartBridge\<MAC\>)
- Device certs (control_devices, ketra_devices) are provisioned at factory per production line (`027-5388-XX`)

### 2. Designer Project Creation
- Designer generates a fresh EC P-256 keypair → becomes `Lutron Project SubSystem Certificate Authority`
- This CA signs:
  - `DesignerClient` cert (for LEAP write access)
  - `RadioRa3Processor<MAC>` cert (stored in DB, sent to processor)
  - SSH user certificates (for root shell access to processor)

### 3. App Pairing (button-press)
- User presses button on bridge/processor
- Processor generates a client cert signed by its own CA
- Client receives cert + key + CA cert
- For RA3: signed by `RadioRa3Processor<MAC>` (which chains to lutron-root)
- For Caseta: signed by `SmartBridge<MAC>` (self-signed root)
- Custom Lutron OIDs encode device serial and name

### 4. Ketra BLE Commissioning
- Ketra bulbs have factory-provisioned device certs under `ketra-devices` CA
- BLE pairing validates against the trust chain bundles in the app
- Trust chain rotated yearly (2020 → 2021 → current/permanent)
- `orion_poc` chain suggests a new product line in development (valid 2025-2026)

### 5. Azure IoT Provisioning
- Two Azure-specific certs per device type: `control-devices-azure-{01,02}`, `ketra-devices-azure-{01,02}`
- Note the typo in these certs: `ST=Pennsylvannia` (double 'n') in the Azure certs vs correct `ST=Pennsylvania` elsewhere
- These enable cloud telemetry and remote management via Azure IoT Hub

## Revoked Certificates

The app ships a blacklist of 6 device certificates:

| CN | Issuer | Notes |
|----|--------|-------|
| `control_devices_027-5388-03` | control-devices | Production line 03 |
| `control_devices_br_027-5388-01` | control-devices | Bridge variant, line 01 |
| `control_devices_027-5388-01` | control-devices | Production line 01 |
| `ketra_devices_027-5388-01` | ketra-devices | Ketra line 01 |
| `ketra_devices_br_027-5388-01` | ketra-devices | Ketra bridge, line 01 |
| `ketra_devices_027-5388-03` | ketra-devices | Ketra line 03 |

These are factory production line certificates — revoking them means entire batches of devices from lines 01 and 03 had their signing certs rotated (replaced by `-01-A` and `-03-A` variants visible in the BLE chain).

## Observations

1. **Caseta is architecturally different** from RA3/HomeWorks. Caseta bridges are self-signed islands; RA3/HomeWorks chain to a global Lutron root CA.

2. **Designer has root SSH** to all processors in its project. The SSH cert is signed by the project's EC key and grants `root` principal. This means Designer can do anything to the processor's Linux OS.

3. **The residential/commercial client certs are universal** — same cert+key ships with every Designer install. This is a shared secret, not per-installer auth. Anyone with a copy of Designer has these certs.

4. **100-year validity on processor certs** (`RadioRa3Processor<MAC>` expires 2125) — designed to outlive the hardware.

5. **Lutron PEN is 40073** (`1.3.6.1.4.1.40073`). Custom OIDs `.1.2` (serial) and `.1.3` (name) are embedded in client certs.

6. **Orion** is a new product line — POC trust chain valid only 2025-2026, suggesting active development of a next-generation platform.

7. **The "Pennsylvannia" typo** in Azure certs (two n's) is a fun artifact — likely a different team or automated pipeline created those certs with a misspelling.

8. **027-5388 = Lutron facility/product code**. All device certs reference this, with line numbers 01-16. Lines 01 and 03 were revoked and reissued as 01-A and 03-A.
